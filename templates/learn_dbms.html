DBMS Tutorial

1. DBMS Overview

What is a DBMS?

A Database Management System (DBMS) is software for storing and retrieving user data while considering appropriate security measures. It consists of a group of programs that manipulate the database. The DBMS accepts requests for data from an application and instructs the operating system to provide the specific data.

2. DBMS Architecture

The 3-Tier Architecture

The design of a DBMS depends on its architecture. The 3-tier architecture, which is the most widely used, separates the system into three independent modules:

  • Presentation Tier (Client): The top-most level of the application, where users interact directly with the system.
  • Application Tier (Business Logic): This tier contains the functional business logic that drives the application's core processes.
  • Data Tier (Database): At this tier, the database, along with its query processing languages, resides.

3. Data Models

Structuring the Data

Data models define how the logical structure of a database is modeled. The most common is the Relational Model, which uses tables to represent data and the relationships between them. Each table has rows (records) and columns (attributes).

4. Data Schemas

Database Design

A database schema is the skeleton structure that represents the logical view of the entire database. It defines how the data is organized and how the relations among them are associated. It formulates all the constraints that are to be applied on the data.

5. Entity-Relationship (E-R) Model

Conceptual Data Modeling

The E-R model is a high-level data model that defines the conceptual view of a database. It is based on the notion of real-world entities and the relationships between them. E-R diagrams are used to illustrate the logical structure of databases.

6. Normalization

Minimizing Redundancy

Normalization is the process of organizing data in a database to avoid data redundancy, insertion anomaly, update anomaly & deletion anomaly. It involves dividing larger tables into smaller, well-structured tables and defining relationships between them.

7. Transactions & ACID Properties

Ensuring Data Integrity

A transaction is a unit of work performed against a database. Transactions are designed to maintain data integrity through the ACID properties:

  • Atomicity: All changes are performed, or none are.
  • Consistency: The database remains in a valid state.
  • Isolation: Concurrent transactions do not interfere with each other.
  • Durability: Committed changes are permanent.
